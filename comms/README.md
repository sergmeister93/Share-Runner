# comms — a portable, agent-agnostic multi-agent coordination protocol

A drop-in coordination layer for getting **multiple AI agents** — Claude Code,
Codex, Cursor, or a mix — to build software together without colliding, lying to
each other, or losing track of what's true. It is **vendor-neutral by design**:
the protocol is a set of files and rules, not a feature of any one tool.

> Deploying to a new project: copy this folder in, rename it to `comms/`, and run
> `node comms/bin/bind.mjs`. That's it.

---

## The one idea

A multi-agent protocol cannot *prevent* an LLM from misbehaving — agents write
free text to files; any rule is advisory. So this protocol does not rely on
goodwill. It is built so that **the correct path is the easy path, and every
violation is caught by a deterministic check** (`bin/validate.mjs`) rather than by
hoping an agent remembered the ritual.

Two consequences shape everything:

1. **The artifact is the truth, never the report.** When a status file disagrees
   with the actual code/bytes/VCS, the artifact wins and the report is wrong.
   `validate.mjs` enforces this; it is wired into a git `pre-commit` hook so it
   fires for *any* agent or human who commits — no harness required.
2. **Two planes.** A portable **data plane** (the files in this folder) that every
   agent can read and write, and an optional **control plane** (a host harness's
   native task/agent/scheduler tools) that *accelerates* the data plane but is
   never required. Codex and Cursor agents use the files; a Claude Code manager may
   also drive native tools — but it **mirrors every change back to the files**, so
   no agent ever sees a different truth.

## Folder map

```
comms/
  KERNEL.md          THE behavior spec. Every agent reads this. Vendor-neutral.
  PROJECT.md         Per-project binding: mode, lanes, commands, frozen contract.
  roster.jsonl       Self-registered agents (append-only). WHO is in play.
  ledger.jsonl       Task state transitions (append-only). Machine truth.
  STATUS.md          Human dashboard — GENERATED by bin/status.mjs, never hand-edited.
  manifest.json      (no-vcs mode only) file -> hash, so drift is detectable.

  schemas/           The exact shape of a note, a ledger entry, a roster entry.
  templates/         Copy-paste forms: work-order, handoff, verification, retro.
  adapters/          How the kernel binds to a substrate:
    vcs.md             git mode (default) vs no-vcs fallback.
    control-plane.md   how native Task/Agent/Monitor/cron tools map onto the files.
    vendors.md         how CLAUDE.md / AGENTS.md / .cursorrules become thin stubs.

  bin/
    validate.mjs     The enforcer. Deterministic. Exit 0 = clean, non-zero = drift.
    bind.mjs         One-time bootstrap: detect env, write PROJECT.md, install hook.
    status.mjs       Regenerate STATUS.md from ledger + roster.
  hooks/
    pre-commit       Vendor-neutral gate: runs validate.mjs, blocks bad commits.
  stubs/             Generated per-vendor pointer files (CLAUDE.md, AGENTS.md, ...).
```

Node (built-ins only, **zero dependencies**) is the one assumed tool. If your
agents can run `node`, the protocol runs. If they can't, the `.md` rules still
apply by hand — you just lose automated enforcement.

## 60-second deploy

1. `cp -r comms_new <project>/comms` (or rename in place).
2. `cd <project> && node comms/bin/bind.mjs` — detects git/node, writes
   `comms/PROJECT.md`, installs the pre-commit hook, generates vendor stubs.
3. Edit `comms/PROJECT.md`: name the **lanes**, the **frozen contract**, the
   **test/build/run commands**, and the **Definition of Done**.
4. Launch your agents. Each one, first action, appends itself to `roster.jsonl`
   and writes a `session-start` ledger entry. The manager assigns work as
   `work-order` entries. Off you go.

Run `node comms/bin/validate.mjs` any time to check the whole project is honest.

See `KERNEL.md` for the rules every agent follows.
